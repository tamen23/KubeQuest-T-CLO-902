# Repository Map

This page explains the role of the main paths in the repository and highlights
which files are meant to be edited directly.

## Application Source

`sample-app-master/` contains the original Laravel application. It still holds
the PHP source code, routes, migrations, tests, Dockerfile, and the historical
docker-compose setup.

Important files:

- `sample-app-master/Dockerfile` builds the application image used in CI/CD.
- `sample-app-master/app/Http/Controllers/MetricsController.php` exposes app
  metrics for Prometheus.
- `sample-app-master/app/Http/Controllers/DebugController.php` exposes guarded
  demo endpoints for CPU, memory, and crash scenarios.
- `sample-app-master/routes/web.php` and `routes/api.php` wire the app routes.
- `sample-app-master/composer.json` and `composer.lock` define PHP
  dependencies.

## Application Manifests

`crementation/` is the Helm chart for the Laravel app. This is the source of
truth for the app Deployment, Service, Ingress, HPA, PDB, ServiceMonitor, and
runtime values.

Important files:

- `crementation/values.yaml` controls image tag, replicas, probes, resources,
  env vars, autoscaling, ingress, and update strategy.
- `crementation/templates/deployment.yaml` renders the app Deployment.
- `crementation/templates/service.yaml` exposes the app internally.
- `crementation/templates/ingress.yaml` exposes the app through ingress-nginx.
- `crementation/templates/hpa.yaml` defines horizontal scaling.
- `crementation/templates/pdb.yaml` protects voluntary disruptions.
- `crementation/templates/servicemonitor.yaml` wires app metrics to Prometheus.

`applications/crementation/` is the Kustomize wrapper that renders the chart.
ArgoCD watches `applications/crementation/overlays/prod/`.

## Database Manifests

`applications/mysql/` renders the official Bitnami MySQL chart. The deployment
uses replication mode, with credentials sourced from the `mysql-secret`
Kubernetes Secret generated by External Secrets Operator.

Important files:

- `applications/mysql/kustomization.yaml` declares the MySQL chart render.
- `applications/mysql/secrets/.gitkeep` keeps the local secret directory shape
  without committing real secrets.

## Platform Manifests

`infrastructure/` is the source of truth for cluster-wide components. The
top-level `infrastructure/kustomization.yaml` renders the platform through
Kustomize and Helm.

Important subdirectories:

- `infrastructure/argocd/` contains ArgoCD values and Application manifests.
- `infrastructure/cert-manager/` contains cert-manager values and issuers.
- `infrastructure/ingress-nginx/` contains ingress controller values.
- `infrastructure/dashboard/` contains Kubernetes Dashboard values.
- `infrastructure/monitoring/` contains Prometheus, Grafana, Loki, Alloy,
  dashboards, RBAC, and alert rules.
- `infrastructure/vault/` contains Vault chart values.
- `infrastructure/external-secrets/` contains the ClusterSecretStore and
  ExternalSecret definitions.
- `infrastructure/dex/` and `infrastructure/oauth2-proxy/` contain the SSO
  layer.
- `infrastructure/network-policies/` contains one policy file per protected
  namespace.
- `infrastructure/velero/` contains Velero values and AWS credential mapping.
- `infrastructure/vpa/` contains the app VPA object.
- `infrastructure/nip-io-reconciler/` contains the CronJob that keeps public
  nip.io hosts and Let's Encrypt annotations in place.

## Vendored Charts

`infrastructure/charts/kubernetes-dashboard/` vendors the Kubernetes Dashboard
chart because its upstream Helm repository is no longer reliable for this
project. Other charts are pulled by Kustomize or Helm at build time.

Do not casually edit vendored chart internals. Prefer values files in
`infrastructure/*/values.yaml` unless the chart itself must be patched.

## Backup Resources

`backups/mysql/` contains the app-level MySQL backup story:

- `cronjob.yaml` runs nightly dumps.
- `pvc.yaml` stores dump files.
- `kustomization.yaml` applies the backup resources.

Velero lives separately in `infrastructure/velero/` because it is a
cluster-level platform component, not only an app backup.

## Policy Components

`components/gatekeeper/` contains the Gatekeeper constraint template and
constraint that reject `:latest` image tags for workloads in the app namespace.
This directory is pulled into the infrastructure layer as a Kustomize resource.

## Automation Scripts

`scripts/` contains operator and demo scripts:

- `cluster-up.sh` provisions AWS resources and forms the kubeadm cluster.
- `deploy.sh` deploys the platform, database, app, backups, public hostnames,
  and NetworkPolicies in the intended order.
- `load-test.sh` generates load for autoscaling demos.
- `failure-demo.sh` drives guarded debug endpoints.
- `drain-demo.sh` demonstrates PDB behavior during node drain.
- `zero-downtime-test.sh` checks rolling-update behavior.

These scripts encode operational knowledge. When a manual README command and a
script disagree, inspect the script before changing the deployment process.

## CI/CD

`.github/workflows/ci.yml` validates changes through image build and scan,
manifest renders, kube-linter, Helm lint, NetworkPolicy coverage checks, and
Docker Hub image publishing on `main`.

`.github/workflows/deploy.yml` is a manual workflow that configures cluster
access, seeds Vault, installs ArgoCD, and hands reconciliation to the ArgoCD
Applications.

## Terraform

`terraform/` creates the AWS layer:

- `main.tf` and `compute.tf` define networking, security group rules, EIPs, and
  EC2 instances.
- `variables.tf` defines the four-node lab layout.
- `outputs.tf` exposes the IPs and generated key path used by the scripts.
- `user-data-kubeadm.sh` installs the node prerequisites for kubeadm.

The Kubernetes platform does not live in Terraform. Terraform only creates the
VMs and AWS networking the cluster runs on.
