# Backups

The repository has two backup layers:

- MySQL logical dumps for application data.
- Velero backups for Kubernetes resources and cluster-level restore demos.

These layers complement each other. MySQL dumps are simple and app-focused.
Velero is broader and supports namespace or full-cluster recovery.

## MySQL Dump CronJob

The app-level backup resources live in `backups/mysql/`.

Files:

- `cronjob.yaml`
- `pvc.yaml`
- `kustomization.yaml`

The CronJob runs in the `crementation` namespace:

```text
schedule: 0 2 * * *
```

It uses the `bitnami/mysql:8.0` image, reads database credentials from
`mysql-secret` and `laravel-db`, connects to `mysql-primary`, and writes a
timestamped SQL dump into `/backup`.

Old dump files are deleted after seven days.

The PVC is:

```text
mysql-backup-pvc
storageClassName: local-path
size: 5Gi
```

## MySQL Backup Checks

Check that the resources exist:

```sh
kubectl -n crementation get cronjob mysql-backup
kubectl -n crementation get pvc mysql-backup-pvc
```

Trigger a manual backup:

```sh
kubectl -n crementation create job --from=cronjob/mysql-backup mysql-backup-manual-test
kubectl -n crementation logs job/mysql-backup-manual-test
```

Inspect the backup PVC by launching a temporary debug pod or by checking the
CronJob logs. Keep this inspection operational only; do not commit dump files.

## Velero

Velero is deployed by the platform layer from `infrastructure/velero/`.

Important files:

- `infrastructure/velero/values.yaml`
- `infrastructure/velero/external-secret.yaml`
- `infrastructure/velero/README.md`

Velero uses:

- AWS plugin image `velero/velero-plugin-for-aws:v1.11.0`.
- Existing Kubernetes Secret `velero-aws-creds`.
- S3 bucket `kubequest-velero-092040680793`.
- AWS region `eu-west-3`.
- Daily full-cluster schedule `daily-full`.
- TTL of `168h0m0s`, which is seven days.

The AWS credential secret is generated by External Secrets Operator from Vault
path `secret/aws`.

## Velero Checks

```sh
kubectl -n velero get pods
kubectl -n velero get externalsecret velero-aws-creds
velero backup-location get
velero schedule get
velero backup get
```

If Velero is installed but cannot access S3, check:

```sh
kubectl -n velero logs deploy/velero --tail=100
kubectl -n velero get secret velero-aws-creds
kubectl -n vault exec vault-0 -- vault kv get secret/aws
```

Do not print real credentials during a presentation.

## On-Demand Backups

Full cluster backup:

```sh
velero backup create full-$(date +%F) --wait
```

App namespace only:

```sh
velero backup create app-$(date +%F) --include-namespaces crementation --wait
```

List results:

```sh
velero backup get
```

## Restore Demo

Namespace restore demo:

```sh
velero backup create demo --include-namespaces crementation --wait
kubectl delete ns crementation
velero restore create --from-backup demo --wait
```

After restore, check:

```sh
kubectl -n crementation get pods,svc,ingress,pvc
kubectl -n crementation get externalsecrets
```

The restore story pairs well with the app-level MySQL backup story: Velero
shows Kubernetes recovery, while the CronJob shows application-data export.

## Fresh Cluster Restore

For a fresh cluster restore:

1. Build the new cluster.
2. Install Velero with the same S3 bucket and compatible AWS credentials.
3. Confirm old backups are visible:

```sh
velero backup get
```

4. Restore the selected backup:

```sh
velero restore create --from-backup <backup-name> --wait
```

This is the strongest backup narrative for the defense because it shows that
state is not tied only to one running cluster.
